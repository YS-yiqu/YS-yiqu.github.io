你有没有遇到过这种诡异场景：写了一个 .vbs 脚本，里面带中文，结果双击就报"无效字符"，但你反复检查，语法绝对没问题 😤？

恭喜你，你撞上了 Windows 史上最古老的「编码代沟」之一。

---

## 一句话说清

**VBScript 引擎是个老顽固——它只认系统编码（中文 Windows 就是 GBK）。你用 UTF-8 保存的 .vbs 文件，中文在这位老顽固眼里就是一堆乱码字节。**

就好比你给一个只懂中文的老大爷发英文短信——每个字母他都认识，但连起来就看不懂了。

---

## 根因：两台「翻译机」各自为政

| 角色 | 翻译标准 | 中文"你好"长啥样 |
|------|----------|-----------------|
| 你的编辑器（现代工具） | UTF-8 | `E4 BD A0 E5 A5 BD`（6 字节） |
| VBScript 引擎（老古董） | GBK / CP936 | `C4 E3 BA C3`（4 字节） |

当 VBScript 用 GBK 的标准去读 UTF-8 的文件，就像用中文拼音规则的发音去念英文单词——念出来全是鬼畜 😅。

`800A0408` 这个错误码翻译成人话就是：「我试着读了，但你这写的啥玩意儿？」

---

## 解决思路框架

遇到「老系统只认 A 编码，新工具只输出 B 编码」的问题，核心思路只有三条路：

```
路 1：妥协 → 只用 ASCII，不碰中文
路 2：迂回 → 让能输出 B 编码的工具代为生成
路 3：对抗 → 改老系统的编码行为（通常不现实）
```

这次走的是**路 2**——用 PowerShell 做「翻译」，把 UTF-8 内容转写成 GBK 编码的文件：

```
人类可读的内容
    ↓
PowerShell 告诉系统"请用 GBK 写这个文件"
    ↓
VBScript 引擎用 GBK 读 → 完美匹配 ✅
```

---

## 通用方法论

这个坑不只是 VBScript 有。任何需要**跨编码边界**的场景都要警惕：

- **cmd 批处理** → 控制台代码页和文件编码要一致
- **PHP 老项目** → 可能还在用 GBK，新编辑器默认 UTF-8
- **MySQL 导入导出** → `utf8mb4` vs `gbk` 的乱码经典

**记住一句话：谁读文件，就用谁的编码标准来写。**

遇到「明明没写错但就是报错」时，先怀疑编码。
