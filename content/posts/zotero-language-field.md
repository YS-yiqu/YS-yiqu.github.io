## 背景

写论文时发现 Zotero 引用列表里，英文论文冒出个「张三等」，中文论文反而不写「等」——心态直接炸裂 😅。折腾一圈才发现，这东西跟 CSL 样式有关，但又不仅仅是样式的事。

---

## 现象

在 Word / Google Docs 里用 Zotero 插件插入引用，出现以下诡异行为：

1. 一篇英文论文，参考文献里显示 "Zhang San **等**"（明明是英文论文为啥写「等」？）
2. 把条目 Language 改成 `zh-CN` 了，刷新后还是 "et al."，改了个寂寞
3. 同样一篇文献，不知道什么时候导入了好几次，也分不清现在引用的是哪条

---

## 根因分析

### 核心机制：Language 字段

Zotero 的引用格式化引擎（CSL）在输出「et al.」这个词时，**会先去读每一条文献的 Language 字段**，然后从对应语言的术语表里挑词：

| 条目 Language | CSL 引擎输出 |
|:---:|:---|
| 空 / `en` / `en-US` | `et al.` |
| `zh-CN` / `zh-TW` | `等` |
| `ja` | `ほか` |
| `ko` | `외` |

> 💡 打个比方：CSL 引擎就像一个翻译官，它拿到一篇文献先看"你说什么语言？"，然后从兜里掏出对应的词典开始翻。Language 字段就是文献报给翻译官的那句话。

### 为什么改 Language 也不生效？

这是 Zotero 最坑的地方——**它允许同名文献存在多条**。

想象你下载了同一篇论文三次：浏览器插件自动抓的、从数据库导入的、手动添加的。三次产生了三个条目，标题一毛一样，但内部 ID 不同，各自有各自的 Language 字段。你在库里改的是第一条，Word 里引用指向的却是第三条——当然不会变。

> 🔑 你不是改错了字段，是**改错了对象**。

### 为什么英文论文会被标成中文？

从 CNKI、万方等中文数据库导入时，导入器会自动给所有条目（包括英文的）的 Language 填上 `zh-CN`。这是数据库自身的设计，Zotero 只是如实照抄。

---

## 解决方案

### 第一步：排查是不是改错了对象

排查比修改重要得多——先搞清楚你到底引用了哪条：

1. 在 Zotero 搜索框搜这篇文献的标题
2. 看看搜索结果是不是有 **2 条以上** 同名条目
3. 回 Word/Google Docs → 点那条引用 → Zotero 插件 → **Edit Citation** → 看当前引用指向的是哪条
4. 确认后去改**被引用的那条**的 Language

### 第二步：合并重复条目

与其修修补补，不如从根上干掉重复：

1. 在 Zotero 里 **Ctrl/Cmd 点击** 选中所有同名条目
2. 右键 → **合并条目 (Merge Items)**
3. Zotero 自动合并元数据 + 附件 + 笔记
4. 回文档 **Refresh**，所有引用自动指向合并后的那条

从此改一次 Language，一劳永逸。

### 第三步：养成习惯

- **导入后立刻检查 Language 字段**（右侧 Info 面板）
- **列表视图加上 Language 列**（右键列标题 → Language），批量扫一眼
- 定期用 Zotero 内置的「检查重复项」清理

---

## 经验教训

1. **Zotero 允许同名不同条**——这是设计，不是 bug。排查时先确认"我到底在编辑哪条"比直接动手改重要得多。

2. **Language 字段是所有「等/et al.」相关问题的唯一真相来源**。不要先去折腾样式文件、不要装插件、不要改 locale，先看一眼 Language。

3. **CSL 样式是"最后一公里"**。样式文件决定的是「term 长什么样」，但用哪个 term，由 Language 字段决定。

4. **相同的排查框架适用于所有多语言引用问题**——不只是「等」，日文「ほか」、韩文「외」同理。
